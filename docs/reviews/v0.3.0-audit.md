## 4. Identidad git (regla de un único autor)

### Comando canónico

```bash
git log --all --format='%an <%ae>' | sort -u
```

### Resultado esperado y aceptado

El comando devuelve dos líneas, ambas con el mismo `Author Name`:

```
GONZALO GARCÍA LAMA <72883726+GonxKZ@users.noreply.github.com>
GONZALO GARCÍA LAMA <gongarlam@alum.us.es>
```

La línea con el correo `72883726+GonxKZ@users.noreply.github.com` corresponde **exclusivamente** a merges efectuados desde la interfaz web de GitHub (botón "Merge pull request"). GitHub firma automáticamente esos commits con el correo público no-reply asociado a la cuenta `GonxKZ` (la del autor). El nombre permanece estable (`GONZALO GARCÍA LAMA`).

Se ha verificado que **no existen** commits con:

- `Co-Authored-By` (cualquier valor).
- Trailers que mencionen `Claude`, `Anthropic`, `Copilot`, `IA`, `AI`, `Bot`, `Assistant`, `GPT`.
- Iconos `🤖` o similares.
- Otros autores con email distinto.

### Política reforzada en `CONTRIBUTING.md`

A partir de M9 la sección "Regla de un único autor en commits y PRs" obliga a configurar la identidad local con `gongarlam@alum.us.es` antes del primer commit y prohíbe coautores y firmas externas. Las PRs que vulneren esta regla se rechazan automáticamente.

## 8. Riesgos abiertos y plan M10+

| Id | Severidad | Descripción | Plan |
|---|---|---|---|
| RA-01 | Baja | 69 warnings `react-doctor` sobre exports de barril. | Limpieza incremental cuando se consuman; sin bloqueo. |
| RA-02 | Media | `recharts` y `maplibre-gl` aún no se cargan con `React.lazy`. | Issue en cola para M10 al introducir el comparador. |
| RA-03 | Baja | `Sidebar.layerState` mantiene patrón uncontrolled. | Conectar al store global en M10 al unificar comparador y mapa. |
| RA-04 | Baja | `pip-audit` informativo. | Política delegada a Dependabot + `SECURITY.md`. |
| RA-05 | Media | Las capturas pixel-perfect dependen de un pase manual cada vez que cambia el lenguaje visual. | Automatizar con baseline visual en `ci-e2e` (M10). |
| RA-06 | Baja | Identidad noreply en merges desde la UI web. | Aceptado: el `Author Name` es estable y la regla académica se cumple. |
| RA-07 | Media | Cobertura geográfica nacional limitada a 101 municipios. | Plan M10 para sumar municipios > 50 000 habitantes y cabeceras de comarca adicionales. |
| RA-08 | Media | Telemetría OpenTelemetry sin desplegar. | Plan M10 para producción real. |

### Plan M10+

1. Sustituir el dataset nacional mock del frontend (`apps/web/src/data/national_mock.ts`) por respuesta real de `/rankings`.
2. Lazy-load de `recharts` y `maplibre-gl`.
3. Subir el umbral de cobertura del frontend de 60 % a 80 % en `vitest.config.ts`.
4. Capa OpenTelemetry para medición real de tiempos de scoring/SPARQL en producción.
5. Pruebas visuales automatizadas con baselines en `ci-e2e`.
6. Persistencia de scores en el grafo RDF para responder `top_scores_by_profile` con SPARQL puro.
7. Multi-tenant ligero (perfiles guardados por usuario) si se decide ampliar más allá del MVP académico.

## 9. Comandos de verificación reproducibles

Para auditar un clon limpio:

```bash
# 1. Identidad git
git log --all --format='%an <%ae>' | sort -u

# 2. Backend
make bootstrap
pytest apps/api/tests
ruff check apps/api/src apps/api/tests
mypy apps/api/src

# 3. Frontend
pnpm -C apps/web install
pnpm -C apps/web lint
pnpm -C apps/web typecheck
pnpm -C apps/web test
pnpm -C apps/web format:check
pnpm -C apps/web build

# 4. RDF
python -c "from rdflib import Graph; Graph().parse('ontology/atlashabita.ttl', format='turtle')"

# 5. SHACL sobre dataset seed
python -c "from rdflib import Graph; from pyshacl import validate; data=Graph(); data.parse('ontology/atlashabita.ttl', format='turtle'); shapes=Graph().parse('ontology/shapes.ttl', format='turtle'); conforms,_,_ = validate(data, shacl_graph=shapes, inference='rdfs'); print('SHACL OK' if conforms else 'SHACL FAIL')"

# 6. E2E
pnpm -C apps/web exec playwright install chromium
pnpm -C apps/web e2e
```

Todos deben terminar con código de salida 0 sobre `develop`.
